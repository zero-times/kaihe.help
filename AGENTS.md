# AGENTS.md

## 项目名称

开盒应对网站（V0 验证版）

## 项目目标

这是一个面向“被开盒 / 被网暴后第一小时”的轻量网站。

当前阶段的目标不是做完整产品，而是用最低成本验证以下问题：

1. 是否存在真实的自然搜索需求。
2. 用户是否会搜索并进入这类页面。
3. 用户进入后，是否愿意阅读并执行页面中的步骤。
4. 哪类页面最值得继续扩写。
5. 这个方向更适合继续做网站，还是后续再考虑小程序 / App。

## 当前阶段定位

当前项目是一个**低成本验证站**，不是正式商业产品，也不是完整工具平台。

### 当前阶段应该做的事

* 建立一个可被搜索、可被分享、可在手机上快速阅读的静态网站。
* 围绕“开盒后的第一小时处理”输出清晰、可执行的页面。
* 优先覆盖最强需求问题，而不是追求内容面面俱到。
* 通过搜索流量、页面停留、转发行为来验证需求强度。

### 当前阶段不做的事

* 不做账号系统。
* 不做复杂交互工具。
* 不做 OCR / 自动识别。
* 不做云端证据托管。
* 不做代举报、代报警、代维权。
* 不做付费咨询。
* 不做重品牌官网。
* 不做 App / 小程序首发版本。

## 目标用户

当前优先服务以下两类用户：

1. 有公开表达、但没有专业支持团队的普通用户，例如学生、小创作者、社交平台活跃用户。
2. 正在帮助家人、伴侣、朋友处理被开盒问题的人。

## 站点核心价值

这个网站要解决的问题不是“解释什么是开盒”，而是：

> 当用户已经遭遇或怀疑遭遇开盒时，能不能在最慌的 30 分钟内知道先做什么、先别做什么。

所以所有页面都要围绕“可执行性”展开，而不是围绕概念解释展开。

## 技术栈

当前阶段建议采用**极简、低成本、低维护**的静态站技术栈，优先保证内容上线、可索引和移动端可读。

### 推荐技术栈

* **托管平台**：GitHub Pages
* **域名**：`kaihe.help`
* **站点框架**：Astro
* **语言**：TypeScript
* **样式方案**：原生 CSS 或轻量 CSS 方案
* **内容来源**：Markdown
* **统计分析**：Google Analytics 或 Plausible（按成本选择）
* **SEO 基础文件**：`sitemap.xml`、`robots.txt`
* **版本管理**：Git + GitHub

### 为什么推荐这套技术栈

* GitHub Pages 几乎零托管成本，适合当前验证阶段。
* Astro 适合内容型网站，天然输出静态 HTML，对 SEO 更友好。
* Markdown 方便快速写作、修改、扩页。
* 原生 CSS 足够支持第一版，不需要为了验证版引入高维护样式系统。
* TypeScript 不是因为项目复杂，而是为了让配置、组件和构建更稳定。

### 当前阶段不推荐的技术方向

* 不推荐一开始使用重型 React SPA。
* 不推荐依赖复杂后端。
* 不推荐引入数据库。
* 不推荐为了验证版搭建 CMS。
* 不推荐为了交互效果增加高维护成本依赖。

## 开发与部署原则

### 开发优先级

1. 页面可访问
2. 页面可读
3. 页面可索引
4. 移动端体验清晰
5. 维护成本低

### 部署原则

* 首发部署到 GitHub Pages。
* 必须绑定自定义域名 `kaihe.help`。
* 必须开启 HTTPS。
* 每次合并到主分支后自动部署。
* 发布前至少检查首页和核心内容页是否可正常访问。

## 内容策略

### 内容原则

每一页都要满足以下要求：

* 标题直接对应用户会搜索的问题。
* 开头先给结论，不绕弯子。
* 主体按步骤展开。
* 明确告诉用户现在先做什么。
* 补充“哪些先别做”或“常见误区”。
* 页面底部给出 2-3 个相关推荐。

### 文风要求

* 使用短句。
* 优先写动作，不优先写背景。
* 避免空泛抒情。
* 避免过度情绪化表达。
* 避免写成法律论文或品牌宣传稿。
* 语气保持冷静、可信、克制。
* 不制造恐慌，但要给出清晰步骤。

### 禁止出现的表达

* “我们提供代维权服务”
* “我们可以帮你报警”
* “保证删帖”
* “官方指定”
* “百分百有效”
* 任何可能构成经营性服务承诺或法律效果承诺的话术

## 第一版内容范围

第一版只做以下页面：

1. 首页
2. 被开盒了怎么办
3. 手机号被公开了怎么办
4. 家人信息被挂出来怎么办
5. 被网暴 / 开盒后先报警还是先举报
6. 12377 怎么举报
7. 开盒后的 30 分钟处理清单
8. 证据怎么留才不容易漏
9. 给家人 / 朋友的提醒模板

原则：先把最关键页面做完整，再决定是否扩页。

## 首页要求

首页不是品牌介绍页，而是答案入口页。

首页必须做到：

* 用户一进来就知道这个站是干什么的。
* 用户一进来就能看到“先做这 4 步”。
* 用户一进来就能根据情境进入对应页面。
* 首页尽量少讲故事，优先给解决路径。

推荐首页主标题：

`被开盒了怎么办｜第一小时处理指南`

## SEO / GEO 原则

本项目第一阶段非常依赖搜索，因此所有内容都必须兼顾 SEO 与 AI 可引用性。

### SEO 基础要求

* 每页必须有独立 `title`。
* 每页必须有独立 `meta description`。
* 每页必须只有一个 `H1`。
* URL slug 要简洁、稳定、可读。
* 页面必须能被搜索引擎抓取。
* 核心内容不能放在图片里。
* 页面之间要有合理互链。
* 站点要有 `sitemap.xml` 和 `robots.txt`。

### GEO / 大模型可引用性要求

* 页面前 200-400 字要直接回答问题。
* 结构尽量清晰，便于抽取。
* 多使用问题-回答、步骤、清单等结构。
* 不要堆砌空洞品牌话术。
* 可为合适页面补 FAQ 结构化数据。

## CI/CD 与自动部署

### GitHub Actions 部署

使用 GitHub Actions 实现"合并到主分支后自动部署到 GitHub Pages"：

* 工作流文件放在 `.github/workflows/deploy.yml`。
* 触发条件：`push` 到 `main` 分支。
* 构建步骤：安装依赖 → `astro build` → 部署到 GitHub Pages。
* 仓库设置中，Pages 的 Source 选择 `GitHub Actions`。

### CNAME 配置

* 在 `public/` 目录下放置 `CNAME` 文件，内容为 `kaihe.help`。
* 这样每次构建部署后域名绑定不会丢失。

## Markdown Frontmatter 规范

每个内容页的 Markdown 文件头部必须包含以下 frontmatter：

```yaml
---
title: "页面标题（会用作 <title> 和 H1）"
description: "页面的 meta description，控制在 120-160 字符"
pubDate: "2026-03-19"
updatedDate: "2026-03-19"
relatedPages:
  - slug: "related-page-slug"
    title: "相关页面标题"
  - slug: "another-related-page-slug"
    title: "另一个相关页面标题"
---
```

* `title` 和 `description` 用于 SEO 和 Open Graph。
* `pubDate` 为首次发布时间，`updatedDate` 为最后更新时间（页面底部显示）。
* `relatedPages` 用于自动生成页面底部的相关推荐。

## 文件与目录建议

建议使用以下结构：

```text
/
  .github/
    workflows/
      deploy.yml
  public/
    CNAME
    favicon.ico
    robots.txt
  src/
    content/
      pages/
        index.md
        what-to-do-after-doxxing.md
        phone-number-exposed.md
        family-info-exposed.md
        report-or-police-first.md
        how-to-report-12377.md
        first-30-minutes-checklist.md
        evidence-checklist.md
        family-message-template.md
    layouts/
      BaseLayout.astro
      ArticleLayout.astro
    components/
      Header.astro
      Footer.astro
      RelatedPages.astro
      StepList.astro
    styles/
      global.css
  astro.config.mjs
  package.json
  tsconfig.json
  AGENTS.md
  CHECKLIST.md
```

说明：

* `sitemap.xml` 由 Astro 的 `@astrojs/sitemap` 集成自动生成，不手动维护。
* `BaseLayout.astro` 处理 HTML 骨架、meta 标签、Header/Footer。
* `ArticleLayout.astro` 继承 BaseLayout，处理内容页的通用结构（标题、更新时间、相关推荐）。
* 组件粒度保持最小够用，不过度抽象。

如果实际实现有所调整，也必须保证：

* 内容文件结构清晰
* 路由语义清楚
* 公共组件尽量复用
* 不引入不必要复杂度

## 页面模板规范

每个内容页尽量使用统一模板：

1. 标题
2. 一句话结论
3. 先做什么
4. 分步骤展开
5. 哪些先别做 / 常见误区
6. 相关推荐
7. 更新时间

## 协作规则

后续无论是人类协作者还是自动代理，在新增页面或修改内容时，都应遵守以下原则：

* 优先保留”先给答案、后给解释”的结构。
* 优先保留”步骤化、可执行”的写法。
* 任何新增内容都要能回答”这个页面为什么值得用户搜到”。
* 不为了扩内容而扩内容。
* 不在第一阶段引入高维护成本方案。

### 内容审核流程

每个页面上线前需经过以下检查：

1. **内容自查**：对照页面模板规范，确认标题、结论、步骤、误区、相关推荐齐全。
2. **风险检查**：检查是否出现禁止表达列表中的措辞，确认无法律承诺、无商业暗示。
3. **工程检查**：构建通过，页面可访问，手机端正常，内部链接无断链。
4. **交叉阅读**：至少以”刚被开盒的慌张用户”视角通读一遍，确认信息足够直接可执行。

## 无障碍基础要求

当前阶段不追求 WCAG AAA，但需满足以下基线：

* 正文字号不小于 16px。
* 主要文字与背景色对比度不低于 4.5:1。
* 所有可点击元素有明确的视觉反馈。
* 页面结构使用语义化 HTML 标签（`<main>`、`<nav>`、`<article>`、`<section>`）。
* 图片提供 alt 文本。
* 页面在纯文本模式下信息仍可理解。

## 性能基线

面向用户场景（紧急、可能用手机流量访问），页面性能非常重要：

* 首页首次加载体积目标 < 200KB（含 HTML + CSS + 图片）。
* 不引入前端 JavaScript 框架运行时（Astro 默认零 JS，保持这一点）。
* 不使用大体积 Web 字体（优先使用系统字体栈）。
* 图片优先使用 SVG 或压缩后的 PNG/WebP。
* Lighthouse Performance 分数目标 > 90。

## Definition of Done

以下条件满足后，相关页面或功能才算完成：

### 内容完成

* 标题明确
* 开头结论明确
* 至少包含可执行步骤
* 包含“先别做什么”或常见误区
* 挂出相关推荐
* 无明显错别字或占位文案

### 工程完成

* 页面可正常访问
* 手机端显示正常
* title 和 meta description 已配置
* 内部链接无断链
* 构建通过
* 部署后线上可访问

## 成功标准

第一阶段不看功能复杂度，主要看以下信号：

* 是否开始出现自然搜索访问。
* 哪些页面更容易获得点击。
* 用户进入后是否会继续阅读。
* 哪些页面更容易被分享。
* 用户更关心总入口页，还是更关心具体问题页。
* “30 分钟清单”类页面是否明显更有吸引力。

## 风险边界

### 内容边界

* 不伪装成官方机构。
* 不提供法律承诺。
* 不声称可替代报警、律师或平台处理。
* 不收集不必要的敏感个人信息。
* 不鼓励对抗升级。

### 产品边界

* 这是一个指南站，不是执法工具。
* 这是一个内容验证站，不是正式应急系统。
* 当前目标是验证需求，不是建立完整产品护城河。
